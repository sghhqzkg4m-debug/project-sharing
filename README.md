# Project Sharing

这个仓库用于分享可安装的 Codex skills。目前包含一个面向产品需求分析和交互设计知识沉淀的 skill：

- `prd-to-knowledge`

## PRD to Knowledge

`prd-to-knowledge` 用于把模糊需求、几句话想法、截图、零散笔记或已有 PRD，整理成更成熟的需求文档与交互设计知识文件。

它适合两类场景：

1. **需求孵化**
   - 输入还不完整，只有几句话、脑图、截图或初步想法。
   - Skill 会先整理假设、补齐用户场景、功能范围、页面模块、状态规则和待确认问题。
   - 输出一份可调整的 V0 需求文档。

2. **PRD 拆解**
   - 输入已有 PRD、产品规格、长文档或半结构化需求。
   - Skill 会拆成面向交互设计师使用的模块化 Markdown 知识文件。
   - 重点沉淀用户任务、入口出口、页面关系、状态反馈、异常恢复、权限差异和设计交付要点。

## 适合输入

- 模糊产品想法
- 几句话需求
- PRD / 用户需求文档
- 截图、脑图、流程说明
- 会议纪要或零散需求笔记
- 需要给交互设计师继续细化的功能需求

## 典型输出

### 需求孵化模式

输出类似：

- 当前理解
- 已明确 / 推断 / 待确认
- 背景与目标
- 用户与场景
- 功能范围
- 页面 / 模块拆解
- 用户流程
- 信息架构与数据对象
- 交互、状态与反馈
- 权限、规则与约束
- UX/UI 设计输入
- 验收标准草案
- 待确认问题
- 建议优先调整项

### PRD 拆解模式

输出类似：

- 模块摘要
- 背景与目标
- 用户、场景与任务
- 范围与边界
- 信息架构
- 用户流程
- 页面与交互拆解
- 交互规则与状态
- 权限、规则与约束
- 内容与文案要点
- 视觉辅助
- 设计产出建议
- 研发协作提示
- 验收标准
- 待确认问题

## 如何调用

安装或启用该 skill 后，可以在 Codex 中这样使用：

```text
$prd-to-knowledge 帮我把这几句话需求细化成 V0 需求文档
```

```text
$prd-to-knowledge 帮我把这个 PRD 拆成给交互设计师使用的知识文件
```

```text
$prd-to-knowledge 基于这张脑图和已有迭代文档，梳理下一版本需求
```

## 使用建议

- 如果需求还不清楚，直接给一句话、截图或脑图即可，不需要先写完整 PRD。
- 如果已有 PRD，建议同时说明目标读者，例如“给交互设计师使用”。
- 如果希望产出文件，可以明确指定目录，例如 `0_Knowledge/` 或某个 Obsidian 项目目录。
- 对不确定内容，skill 会用 `推断` 和 `待确认` 标记，方便后续产品、设计、研发一起调整。

## 维护说明

- `SKILL.md` 保存触发描述和主工作流。
- `references/refined-prd-format.md` 保存模糊需求孵化为 V0 PRD 的模板。
- `references/output-format.md` 保存 PRD 拆解为交互设计知识文件的模板。
- `agents/openai.yaml` 保存展示名、简短说明和默认提示词。

内部调用名请保持为：

```yaml
name: prd-to-knowledge
```

不要改成中文名，否则可能影响 `$prd-to-knowledge` 的稳定调用。
